# CM93 vector charts

The app reads **C-Map CM93 version 2** vector charts in addition to ENC (S-57).
CM93 support is delivered as a **dynamically-loaded plugin**, for both
architectural and licensing reasons.

> **The plugin source lives in a separate repository** (`chartplotter-cm93`),
> because it is GPL-2.0 (see below). This app's tree contains only the LGPL
> plugin-SDK headers it builds against (`src/chart_source.hpp` and friends).
> File paths below of the form `cm93_*.cpp` refer to that separate repo.

## Why a plugin (licensing)

CM93 is a proprietary binary format. The only practical reference for decoding it
is [OpenCPN](https://github.com/OpenCPN/OpenCPN)'s `cm93.cpp`, which is
**GPL-2.0-or-later**. The host application is **LGPL-2.1**. Porting CM93 decoding
into the host would relicense the whole app to GPL.

Instead the CM93 reader lives entirely in `plugins/cm93_plugin/` (GPL-2.0), built
as a separate module and loaded at runtime via `QPluginLoader`. The host links
none of it at build time and talks to it only through the LGPL chart-source ABI
(`src/chart_source.hpp`). The host remains LGPL and still builds/runs as an
ENC-only program without the plugin. See `plugins/cm93_plugin/README.md` and
`plugins/cm93_plugin/COPYING`.

## The chart-source seam

The whole pipeline downstream of a parsed cell — quilting, the `FeatureCache`
LRU, clip/build, S-52 symbology, paint — is keyed on two value types the plugin
boundary already carries: `Feature` (projected geometry + S-57 acronym + attrs)
and a per-cell footprint. So CM93 plugs in by producing exactly what the built-in
ENC reader produces:

| Built-in ENC path (default) | `IChartSource` method (plugin) |
|---|---|
| `ChartCatalog` enumerates `*.000` + `computeCellCoverage` | `catalog(root)` → cells: id, band, bbox, coverage |
| `chart::loadCellFeatures` (GDAL S-57) | `loadCell(id)` → `std::vector<Feature>` |
| `bandFromPath` (filename digit, 6 ENC bands) | source maps its native scale → band |

`ChartCatalog` and `ChartView` each take an optional `IChartSource*` (null =
built-in GDAL path). `MainWindow::startScan` offers the chart folder to each
registered source's `canHandle()` and uses the first match; otherwise it falls
back to the ENC reader. A CM93 folder is recognised by its `CM93OBJ.DIC`
dictionary (or scale-char cell files).

## How a CM93 cell is decoded

(Ported from OpenCPN into `plugins/cm93_plugin/cm93_decoder.cpp` /
`cm93_dictionary.cpp`.)

1. **Dictionary.** `CM93OBJ.DIC` maps object-class codes to S-57 acronyms
   (DEPARE, SOUNDG, BOYLAT, …); `ATTRLUT.DIC` or `CM93ATTR.DIC` maps attribute
   codes + value types. A few class substitutions are applied (e.g. `ITDARE` →
   `DEPARE`), and `COLMAR` colour enums become S-52 `COLOUR` lists.
2. **Deobfuscation.** Cell files are byte-scrambled through a fixed 256-entry
   table; every field is decoded on read.
3. **Cell layout.** A 10-byte prolog + 128-byte header (with a self-consistency
   length checksum), then four tables: vector edges, 3-D points (soundings),
   2-D points, and feature records.
4. **Geometry.** Points are cell-relative `uint16` pairs de-quantised with the
   header's transform coefficients onto the **International 1924** ellipsoid
   (semi-major axis 6378388 m), minus a per-cell WGS84 offset (from the cell's
   `M_COVR` object), then converted to lon/lat and projected to Mercator for the
   host. Areas assemble multi-ring polygons from shared edges (honouring segment
   direction + ring closure); lines concatenate edges; soundings carry depth.
5. **Mapping.** Each object becomes a `Feature`: classified to a `FeatureKind`,
   tagged with its S-57 `objClass` + attributes, with `OBJNAM`/`SCAMIN`/depth
   pulled out like the ENC loader does.

## Scales and bands

CM93 is a single global database at **8 discrete scales**, coarsest→finest:

| scale | ~native | host band |
|---|---|---|
| Z | 1:20,000,000 (world) | 1 |
| A | 1:3,000,000 | 2 |
| B | 1:1,000,000 | 3 |
| C | 1:200,000 | 4 |
| D | 1:100,000 | 5 |
| E | 1:50,000 | 6 |
| F | 1:20,000 | 7 |
| G | 1:7,500 (harbour) | 8 |

The host band model spans 1..8 (`ChartView::kMaxBand`); ENC uses only 1..6.
**Every** CM93 scale covers the whole world, so each maps to its *own* band:
if two overlapping scales shared a band, the quilt (which treats same-band cells
as non-overlapping tiles) would draw both and flicker as you zoom. Distinct bands
let finer scales occlude coarser ones cleanly, and coarser scales fill the gaps.

## Catalog cost and caching

A full footprint (header bbox + `M_COVR` coverage outline) needs a complete cell
decode, and a worldwide CM93 set is large (~30k cells / ~1.9 GB). So the plugin:

- caches each cell's footprint to disk
  (`AppData/.../cm93_catalog_cache/<sha1(root)>.bin`), keyed by file
  path + size + mtime — only the *first* scan decodes every cell, and only
  changed/new cells are re-decoded afterwards;
- decodes the cold cells **in parallel** (`QtConcurrent`), reporting progress
  through the new `IChartSource::catalog` progress callback so the host's scan
  status advances instead of appearing to hang.

## Limitations / next steps

- `.xz`-compressed cells are skipped (no LZMA decompression yet).
- The scale→band thresholds (`bandForVisibleWidth`) reuse the ENC breakpoints;
  they are monotonic and serviceable but not tuned to CM93's exact native scales.
- M_COVR holes (no-data areas inside a coverage outline) are treated as covered
  (over-covering is the safe direction for quilting, as with the ENC path).
